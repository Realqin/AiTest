# MySQL Schema

## Design Goals

- Move core business query fields out of generic `payload JSON` into explicit relational columns.
- Keep `extra_data JSON` only for low-frequency extension fields to avoid schema churn.
- Preserve the existing `db.xxx` facade so upper-layer API code can continue using dictionary-style records.

## Table Overview

### `id_sequences`

Sequence allocator for business IDs.

| Column | Type | Notes |
| --- | --- | --- |
| `scope` | `VARCHAR(64)` | Sequence scope primary key |
| `current_value` | `INT` | Current allocated numeric value |

Current rule:

- Business IDs are generated by the backend, not by the frontend.
- IDs are zero-padded numeric strings, currently formatted as `000000000001`.
- Each business table uses its own sequence scope, so `projects` and `llm_configs` can both start from `000000000001`.
- The table list pages may display a separate front-end row number; that is not the database ID.

## Structured vs JSON

Structured columns now hold:

- Query, filter, sort, join, and display-critical fields
- Fields that appear directly in list pages or detail headers
- Fields commonly referenced across tables, such as foreign keys and status values

`extra_data JSON` now holds:

- Rarely used compatibility fields
- Future extension fields that do not justify a migration yet
- Low-frequency UI metadata

## Recommended Follow-up

- Add Alembic migrations before moving this schema into shared or production environments.
- Gradually encrypt or externalize sensitive fields such as `llm_configs.api_key`.
- If review result querying becomes frequent, split `review_runs.results` into a child table.
